## 一、用户需求

### 1.1 应用场景与目标用户

#### 应用场景

随着大语言模型（LLM）技术的快速发展，单一智能体在处理复杂研究任务时面临显著的能力瓶颈：信息收集不够全面、分析深度不足、报告质量难以自证。多智能体协作（Multi-Agent Collaboration）通过将复杂任务分解为多个专业子任务，由不同角色的智能体各司其职、协同完成，能够显著提升研究效率和质量。

本系统构建了一个**多智能体协作研究助手**，用户只需输入研究主题，系统自动调度5个专业智能体（编排器、研究员、分析师、写作员、审查员）协作完成从需求分析到质量审查的全流程研究任务，最终输出结构化的研究报告和质量评估。系统支持三种研究模式（快速/标准/深度），深度模式下支持并行研究和反思闭环优化，体现了智能体系统的灵活性、并行性和自我改进能力。

#### 目标用户

| 用户类型 | 典型需求 | 使用场景 |
|---------|---------|---------|
| 高校学生 | 快速了解某研究领域的基础知识和发展脉络 | 课程论文选题、文献综述 |
| 研究人员 | 快速获取某主题的综述性分析 | 课题调研、跨领域探索 |
| 企业决策者 | 快速了解某技术领域的现状与趋势 | 技术选型、投资决策 |
| 知识工作者 | 高效完成信息检索与整理任务 | 行业报告、竞品分析 |

## 三、软件体系风格设计

### 3.1 采用的体系风格

本系统采用**管道-过滤器（Pipe-and-Filter）+ 事件驱动（Event-Driven）**的混合体系风格。

#### 主风格：管道-过滤器（Pipe-and-Filter）

多智能体协作的核心流程采用管道-过滤器风格。每个智能体作为一个**过滤器（Filter）**，接收输入数据，经过LLM处理后输出结果；**管道（Pipe）**即AgentPipeline，负责协调数据在各过滤器间的流转。

**标准模式管道：**
```
[用户输入] → ┌────────┐ → ┌────────┐ → ┌────────┐ → ┌────────┐ → ┌────────┐ → [研究报告]
             │编排器   │   │研究员  │   │分析师  │   │写作员  │   │审查员  │
             │Filter 1│   │Filter 2│   │Filter 3│   │Filter 4│   │Filter 5│
             └────────┘   └────────┘   └────────┘   └────────┘   └────────┘
```

**管道-过滤器关键特性在本系统中的体现：**

| 管道-过滤器特性 | 本系统体现 |
|---------------|-----------|
| 过滤器独立性 | 每个智能体独立封装，有独立的system_prompt和execute方法 |
| 数据流驱动 | 前一步输出作为后一步输入（context传递） |
| 可组合性 | 智能体可自由组合（快速2个、标准5个、深度5个+扩展） |
| 透明管道 | Pipeline不关心过滤器内部逻辑，只负责数据传递和流程控制 |

**深度模式扩展——引入并行分支和反馈回路：**

```
                              ┌──────────┐
                         ┌──→ │研究维度1 │ ──┐
[用户输入]→[编排器]→[Map]─┤    │(并行)    │   │[Reduce]→[分析师]→[写作员]→[审查员]
                         └──→ │研究维度2 │ ──┘              ↑          │
                              └──────────┘                  │          ↓
                                              [反思闭环]←───┘    评分<80?
                                                                       │是
                                                              [写作员重写]→[审查员重评]
```

这种扩展使管道-过滤器风格突破了严格的线性管道限制，形成了**带反馈回路的增强管道**。

#### 辅助风格：事件驱动架构（Event-Driven）

智能体间的状态通知和实时进度推送采用事件驱动风格：

```
[智能体] ──publish──→ [MessageBus] ──notify──→ [SSE Handler] → 前端工作流卡片
                     (Event Channel)   ├──→ [WebSocket Handler] → 前端状态面板
                                       └──→ [日志系统] (可扩展)
```

**事件类型定义：**

| 事件类型 | 触发时机 | 携带数据 |
|---------|---------|---------|
| pipeline_start | 管道启动 | topic, mode, session_id |
| agent_start | 智能体开始工作 | agent_type, task |
| agent_output | 智能体产出结果 | agent_type, content |
| reflection_start | 反思闭环触发 | round, feedback |
| pipeline_complete | 管道完成 | report, review, total_score |
| pipeline_error | 管道异常 | error_message |

### 3.2 风格选择理由

#### 为什么选择管道-过滤器作为主风格？

| 理由 | 详细说明 |
|------|---------|
| **任务天然有序** | 研究流程具有明确的阶段顺序：分解→研究→分析→写作→审查，各阶段存在数据依赖关系（分析师依赖研究员输出，写作员依赖研究和分析结果），管道风格天然适配 |
| **数据流清晰** | 每个阶段的输入输出明确，前一步输出是后一步输入，数据流可追踪、可审计 |
| **过滤器独立性** | 每个智能体独立封装，修改一个智能体的prompt或逻辑不影响其他智能体，符合开闭原则 |
| **可组合性** | 智能体可自由组合形成不同管道：快速模式2个、标准模式5个、深度模式5个+扩展，体现了管道的可重组能力 |
| **可测试性** | 每个过滤器可独立测试，给定固定输入验证输出是否符合预期 |

#### 为什么选择事件驱动作为辅助风格？

| 理由 | 详细说明 |
|------|---------|
| **实时性需求** | 用户需要实时观察智能体工作过程（类似智谱Agent模式、Kimi Agent模式），事件驱动天然支持异步通知 |
| **松耦合** | 智能体不需要知道谁在监听它的状态变化，发布者与订阅者解耦 |
| **可扩展** | 新增监听者（如日志系统、监控系统、统计分析）无需修改智能体代码，只需订阅消息通道 |
| **关注点分离** | 智能体专注于"做研究"，进度推送由事件系统负责，职责清晰 |

#### 为什么采用混合风格而非单一风格？

单一管道-过滤器无法处理以下需求：

| 需求 | 纯管道-过滤器的局限 | 混合风格的解决方式 |
|------|-------------------|------------------|
| 实时进度推送 | 管道只关注数据流转，不关注状态通知 | 事件驱动提供异步通知机制 |
| 反思闭环 | 严格管道是单向的，无法回溯 | 增强管道引入反馈回路 |
| 并行研究 | 严格管道是串行的 | 增强管道引入并行分支（Map-Reduce） |
| 多端监听 | 管道无广播能力 | 事件驱动支持一对多通知 |

**混合风格兼顾了流程的有序性（管道保障）和系统的灵活性（事件驱动保障）。**

### 3.3 优缺点分析

#### 管道-过滤器风格

| 优点 | 缺点 |
|------|------|
| 各智能体独立开发测试，修改互不影响 | 严格串行可能成为性能瓶颈（标准模式） |
| 数据流清晰可追踪，便于调试和审计 | 过滤器间数据格式需统一（均使用字符串+JSON） |
| 新增智能体只需接入管道，符合开闭原则 | 难以处理需要回溯的场景（需引入反馈回路） |
| 支持不同组合模式（快速/标准/深度），灵活可配 | 管道协调逻辑随模式增加而复杂化 |
| 每个过滤器可独立替换LLM或调整prompt | 数据在管道中全量传递，可能存在冗余 |

#### 事件驱动风格

| 优点 | 缺点 |
|------|------|
| 松耦合，智能体与展示层完全解耦 | 事件流可能难以调试（异步执行顺序不确定） |

## 四、智能体设计模式分析

本系统综合运用了6种智能体设计模式，覆盖了从输入安全到任务编排、从并行处理到质量保证的全链路。

### 4.1 ReAct模式（推理+行动）

#### 模式描述

ReAct（Reasoning + Acting）模式让智能体交替进行推理（Thought）和行动（Action），通过"思考→行动→观察→再思考"的循环来完成任务。相比纯推理（Chain-of-Thought）或纯行动（Act-only），ReAct能够更好地将推理与决策结合。

#### 本系统应用

**编排器智能体**采用ReAct模式：

- **推理（Reasoning）**：分析用户研究需求，理解任务的核心目标、研究范围和约束条件
- **行动（Acting）**：将需求分解为结构化的任务计划（JSON格式），包含研究重点、分析维度、写作要求、质量标准
- **观察（Observation）**：接收任务计划的执行反馈，为后续智能体提供指引

```
用户输入: "人工智能在医疗领域的应用"
    ↓
[推理] 分析需求：这是一个跨学科研究主题，需要从技术、应用、伦理等多维度分析
    ↓
[行动] 输出任务计划JSON:
  {
    "task_summary": "研究AI在医疗领域的应用现状与发展趋势",
    "research_focus": "AI在医疗的应用与前景",
    "analysis_dimensions": ["技术现状", "应用场景", "伦理挑战"],
    "writing_requirements": "结构清晰、逻辑连贯、数据支撑",
    "quality_criteria": "信息准确、分析深入、建议可行"
  }
    ↓
[观察] 任务计划传递给后续智能体执行
```

**代码实现关键**（orchestrator.py）：
- 使用低温度（0.3）保证输出稳定性和JSON格式一致性
- 内置JSON解析容错：支持提取 ```json 代码块中的内容
- 解析失败时提供默认降级方案（3个通用分析维度）

#### 作用

- 将模糊的用户需求转化为精确的可执行任务计划
- 为后续智能体提供明确的执行指引，减少理解偏差
- 提高系统的可控性和可预测性，避免智能体"自由发挥"

### 4.2 反思闭环模式（Reflection Loop）

#### 模式描述

反思模式让智能体对自身输出进行评估，若不满足质量标准则触发迭代优化，形成"生成→评估→反馈→重新生成"的闭环。这是保证输出质量的关键机制，模拟了人类专家的"审稿-修改"工作流。

#### 本系统应用

**审查员+写作员**构成反思闭环：

- 审查员从5个维度评估报告质量（各20分，满分100）：
  1. 内容完整性（Content Completeness）
  2. 逻辑连贯性（Logical Coherence）
  3. 信息准确性（Information Accuracy）
  4. 分析深度（Analysis Depth）
  5. 表达清晰度（Expression Clarity）
- 若总分低于80分阈值，自动触发写作员根据审查反馈重写
- 最多迭代2轮，防止无限循环

```
[写作员生成报告] → [审查员评估: 72分 < 80分]
    ↓
[反思触发] 审查反馈: "分析深度不足(14/20), 建议增加数据支撑; 逻辑连贯性待提升(15/20)"
    ↓
[写作员根据反馈优化] → [审查员重新评估: 85分 ≥ 80分]
    ↓
[反思结束] 输出最终报告
```

**代码实现关键**（pipeline.py）：
```python
# 反思闭环核心逻辑
max_reflection_rounds = 2
for round_num in range(1, max_reflection_rounds + 1):
    review_result = await self.reviewer.execute(...)
    total_score = review_result.get("total_score", 0)
    if total_score >= 80:
        break  # 质量达标，退出反思
    # 将审查反馈注入写作员上下文，触发重写
    writing_requirements = f"根据审查反馈优化报告：{review_result.get('suggestions', '')}"
    writer_result = await self.writer.execute(...)
```

#### 作用

- 保证输出质量，避免低质量报告直接交付用户
- 模拟人类专家的"审稿-修改"工作流，体现智能体自我改进能力
- 反馈信息指导优化方向，使每轮迭代有明确目标

### 4.3 Map-Reduce模式（并行聚合）

#### 模式描述

Map-Reduce模式将任务分解为多个可并行的子任务（Map），分别处理后聚合结果（Reduce），适用于可分解的批量处理场景。在智能体系统中，该模式可显著提升处理效率。

#### 本系统应用

**研究员智能体**在深度模式下采用Map-Reduce：

- **Map阶段**：根据编排器输出的分析维度，并行启动多个研究任务（asyncio.gather并发执行）
- **Reduce阶段**：将各维度的研究结果按维度标题汇总为综合研究文档

```
编排器输出: analysis_dimensions = ["技术现状", "应用场景", "伦理挑战"]
    ↓
[Map] 并行研究 (asyncio.gather):
  ├── 研究"技术现状" → 结果1（800-1200字）
  ├── 研究"应用场景" → 结果2（800-1200字）
  └── 研究"伦理挑战" → 结果3（800-1200字）
    ↓
[Reduce] 汇总:
  综合研究文档 = "# 主题\n## 技术现状\n结果1\n## 应用场景\n结果2\n## 伦理挑战\n结果3"
```

**代码实现关键**（pipeline.py）：
```python
# Map-Reduce并行研究
if mode == "deep" and len(dimensions) >= 2:
    # Map: 并行研究各维度
    tasks = [self.researcher.execute(
        task=f"围绕'{topic}'主题，重点研究'{dim}'维度",
        context={"research_focus": research_focus, "analysis_dimension": dim}
    ) for dim in dimensions]
    results = await asyncio.gather(*tasks)
    # Reduce: 汇总各维度研究结果
    research_data = "# 综合研究报告\n\n"
    for dim, result in zip(dimensions, results):
        research_data += f"## {dim}\n{result}\n\n"
```

#### 作用

- 提升研究效率，多维度并行研究节省时间
- 保证各维度的研究深度，避免单一研究员信息遗漏
- 体现智能体系统的并行处理能力，充分利用异步I/O

### 4.4 路由分支模式（Routing/Branching）

#### 模式描述

路由模式根据输入特征或用户选择，将请求分发到不同的处理路径，实现灵活的流程控制。这是智能体系统中实现"按需调度"的核心模式。

#### 本系统应用

**AgentPipeline**根据用户选择的研究模式进行路由：

| 模式 | 智能体链 | 特点 | 适用场景 |
|------|---------|------|---------|
| 快速(quick) | 编排器→写作员→审查员 | 2个智能体，跳过研究和分析 | 快速了解、时间紧迫 |
| 标准(standard) | 编排器→研究员→分析师→写作员→审查员 | 5个智能体串行 | 一般研究、平衡效率与质量 |
| 深度(deep) | 5个智能体+并行研究+反思闭环 | 最完整流程 | 深度研究、质量优先 |

```
用户选择模式
    ├── quick  → [编排器] → [写作员] → [审查员]
    │                        ↑ 直接基于编排结果写作
    ├── standard → [编排器] → [研究员] → [分析师] → [写作员] → [审查员]
    │                                    ↑ 串行执行
    └── deep → [编排器] → [研究员(并行Map-Reduce)] → [分析师] → [写作员] → [审查员]
                │                                                    ↑          │
                │                                               [反思闭环? ←────┘]
                └── [观察者模式: 事件广播] ← 所有智能体
```

### 4.5 观察者模式（Observer/Pub-Sub）

#### 模式描述

观察者模式定义对象间的一对多依赖关系，当一个对象状态改变时，所有依赖它的对象都会收到通知。发布-订阅（Pub-Sub）是观察者模式的进阶变体，通过事件通道解耦发布者和订阅者。

#### 本系统应用

**MessageBus**实现发布-订阅模式：

- 智能体通过`message_bus.publish(channel, message)`发布状态变更和结果消息
- SSE端点、WebSocket端点等订阅消息通道，实时推送给前端
- 新增监听者无需修改智能体代码，符合开闭原则

```
[智能体] ──publish──→ [MessageBus] ──notify──→ [SSE Handler] → 前端工作流卡片
                     (通道: agent_researcher)  → [WebSocket Handler] → 前端状态面板
                                              → [日志系统] (可扩展)
                                              → [监控系统] (可扩展)
```

**代码实现关键**（message_bus.py）：
```python
class MessageBus:
    def __init__(self):
        self._subscribers: Dict[str, List[Callable]] = {}
        self._websocket_handlers: List[Callable] = []
        self._history: List[AgentMessage] = []

    async def publish(self, channel: str, message: AgentMessage):
        # 通知订阅者
        for handler in self._subscribers.get(channel, []):
            await handler(message)
        # 通知WebSocket处理器
        for ws_handler in self._websocket_handlers:
            await ws_handler(message)
        # 记录历史
        self._history.append(message)
```

#### 作用

- 实现智能体与展示层的完全解耦，智能体无需知道谁在监听
- 支持实时进度推送，用户体验流畅
- 便于扩展新的消息消费者（日志、监控、统计），零侵入

### 4.6 安全护栏模式（Guardrails）

#### 模式描述

安全护栏模式在系统边界处对输入/输出进行校验和过滤，防止恶意或异常数据进入系统核心逻辑。本系统采用**三层纵深防御**策略，在输入层、LLM层分别部署安全机制。

#### 本系统应用

**第一层：Guardrails模块（API入口层）**——校验用户输入：

| 校验规则 | 检测内容 | 示例 |
|---------|---------|------|
| 空值检查 | 输入不能为空 | "" |
| 长度限制 | 1-2000字 | 超长输入 |
| 提示注入检测(中文) | 忽略/无视/跳过 + 指令/规则/限制 | "忽略之前的指令" |
| 提示注入检测(英文) | pretend/ignore/override/bypass + instruction/rule | "bypass all constraints" |
| 恶意角色扮演检测 | 你(现在/必须/应该) + 是/成为/扮演 + 黑客/恶意 | "你现在是一个黑客" |

**第二层：学术化Prompt设计 + 自动重试（LLM调用层）**——运行时容错：
- 所有智能体的system prompt均采用**学术化定位**（学术研究员、学术分析师、学术写作专家）
- prompt中显式声明【核心原则】：只输出客观中立内容、不臆测不夸大
- 当GLM API返回内容过滤错误（400/contentFilter/1301）时，自动重试最多2次
- 重试时追加安全约束指令到system prompt，并降低温度使输出更稳定

```
用户输入 → [Guardrails校验] → 通过 → [Pipeline执行]
                         → 不通过 → 返回HTTP 400

[Pipeline内部 - BaseAgent._think()]
→ 调用GLM API → 成功 → 返回结果
→ 内容过滤错误(400) → 追加安全约束+降温度 → 重试(最多2次)
→ 其他错误 → 直接抛出异常
```

**代码实现关键**（guardrails.py）：
```python

    # 空值检查、长度限制、提示注入检测（3个正则模式）
    # ...
```

**代码实现关键**（base.py - 自动重试机制）：
```python
async def _think(self, prompt, temperature=0.7, max_retries=2):
    for attempt in range(max_retries + 1):
        try:
            result = await llm_client.chat(messages, temperature)
            return result
        except Exception as e:
            if "400" in str(e) or "contentFilter" in str(e):
                if attempt < max_retries:
                    messages[0]["content"] += SAFETY_INSTRUCTION
                    temperature = min(temperature * 0.8, 0.3)
                    continue
            raise
```

#### 作用

- **两层纵深防御**：入口校验 + LLM层重试，全方位保障安全性
- 防止提示注入攻击和内容安全过滤导致的流程中断
- 体现防御性编程思想，增强系统鲁棒性和可用性

6种设计模式并非孤立存在，而是形成了一个有机协作的整体：

用户请求 → [安全护栏] → [路由分支] → 选择处理路径

#### 代码复用与设计原则

| 设计原则 | 体现 |
|---------|------|
| 开闭原则(OCP) | 新增Agent只需继承BaseAgent，无需修改Pipeline |

| 模板方法 | BaseAgent定义execute骨架，子类实现具体逻辑 |

#### 并发与调度机制

| 机制 | 实现方式 | 说明 |
|------|---------|------|
| asyncio事件循环 | 单线程异步 | 所有I/O操作异步执行，避免线程切换开销 |
| SSE流式推送 | asyncio.Queue + 生成器 | 生产者(Pipeline)入队，消费者(SSE生成器)出队推送 |
| 并行研究 | asyncio.gather() | 深度模式下多维度研究任务并发执行 |
| WebSocket | 独立异步连接 | 与HTTP共享事件循环，双向实时通信 |
| 事件回调 | event_callback异步函数 | Pipeline→SSE的实时事件传递通道 |
| 心跳保活 | SSE 1秒超时 | 防止连接超时断开 |

### 6.3 启动方式
